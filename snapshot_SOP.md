# 🔄 快照存檔標準作業流程（Snapshot SOP）

> **文件說明**：本 SOP 定義每次分析完成後的標準存檔步驟，確保系統持續累積、不漏接任何重要判斷。

---

## ▌觸發條件

以下任一情況發生時，應執行存檔流程：

1. 完成一份新報告的摘要與分析討論後
2. 某個產業或個股的判斷發生明顯改變後
3. 重大市場事件發生後（例如：法說會、重大政策公告）
4. 每週至少執行一次例行存檔（即使無重大更新）

---

## ▌存檔流程（7 步驟）

```
步驟 1：填寫快照文件
  → 複製 snapshot_template.md
  → 填入本次分析的所有欄位
  → 命名為：YYYY-MM-DD_SNAP-XXX.md

步驟 2：更新記憶庫
  → 根據快照中「待更新的記憶庫文件」清單
  → 逐一更新 memory/ 下的對應文件
  → 在歷史紀錄區塊新增條目（不覆蓋舊資料）

步驟 3：更新主控制文件
  → 開啟 00_MASTER.md
  → 更新「系統版本資訊」的日期與快照編號
  → 更新「當前市場定位」摘要
  → 更新「當前產業觀點摘要」表格
  → 更新「當前個股追蹤清單」表格
  → 更新「快照索引」表格

步驟 4：存檔報告摘要
  → 將本次分析的原始報告摘要
  → 存入 reports/archive/ 資料夾
  → 命名格式：YYYY-MM-DD_（報告類型）_（來源）.md

步驟 5：版本控制（GitHub 用戶）
  → git add .
  → git commit -m "SNAP-XXX: （本次更新摘要）"
  → git push

步驟 6：Notion 同步（Notion 用戶）
  → 將更新後的 00_MASTER.md 貼入 Notion 對應頁面
  → 在 Notion 快照資料庫新增本次快照記錄

步驟 7：清理待解決問題清單
  → 回到 00_MASTER.md 的「待解決的關鍵問題」
  → 刪除已解決的問題
  → 新增本次分析產生的新問題
```

---

## ▌命名規範

| 文件類型 | 命名格式 | 範例 |
|----------|----------|------|
| 快照文件 | `YYYY-MM-DD_SNAP-XXX.md` | `2025-03-15_SNAP-012.md` |
| 報告摘要（總經） | `YYYY-MM-DD_macro_（來源縮寫）.md` | `2025-03-15_macro_MS.md` |
| 報告摘要（產業） | `YYYY-MM-DD_industry_（產業）_（來源縮寫）.md` | `2025-03-15_industry_semi_GS.md` |
| 報告摘要（個股） | `YYYY-MM-DD_stock_（代號）_（來源縮寫）.md` | `2025-03-15_stock_2330_法說會.md` |
| 報告摘要（新聞） | `YYYY-MM-DD_news_（主題關鍵字）.md` | `2025-03-15_news_CoWoS供給.md` |

---

## ▌每月例行維護

每月最後一個週末，執行以下維護工作：

1. **彙整月度快照**：回顧當月所有快照，提取最重要的判斷變化，更新 `00_MASTER.md` 的宏觀摘要
2. **清理停止追蹤的個股**：將已停止追蹤的個股從 `stock_tracker.md` 主清單移至「已停止追蹤」區塊
3. **校準論點準確性**：在 `my_thesis.md` 的「論點驗證追蹤」表格中填入已到期判斷的實際結果
4. **更新系統版本號**：在 `00_MASTER.md` 將版本號遞增（例如 v1.0 → v1.1）
